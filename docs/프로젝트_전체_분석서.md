## 1. 프로젝트 개요

**몽글픽(MongglePick)**은 영화 추천 AI 에이전트 시스템이다. 사용자의 자연어 대화를 분석하여 감정/선호를 파악하고, 157,194편의 영화 데이터베이스에서 개인화된 추천을 제공한다.

### 핵심 특징

| 항목 | 내용 |
|---|---|
| **에이전트 아키텍처** | LangGraph StateGraph 기반 2단계 중첩 그래프 |
| **LLM** | Ollama 로컬 서빙 (EXAONE 32B + Qwen 3.5 35B) |
| **임베딩** | Upstage Solar 4096차원 |
| **검색** | 하이브리드 3-way (Qdrant 벡터 + ES BM25 + Neo4j 그래프) → RRF 합산 |
| **추천 알고리즘** | CF + CBF 하이브리드 + MMR 다양성 재정렬 |
| **데이터 규모** | 157,194편 영화, 270,888명 유저, 26M 시청 이력 |
| **이미지 분석** | VLM 기반 포스터/분위기 사진 분석 → 자동 추천 |
| **API** | FastAPI SSE 스트리밍 + 동기 JSON + 멀티파트 업로드 |
| **관측성** | LangSmith 트레이싱 + structlog 구조화 로깅 |

### 구현 완료 Phase

| Phase | 내용 | 상태 |
|---|---|---|
| Phase 0 | 프로젝트 스캐폴딩 | ✅ |
| Phase 1 | 데이터 파이프라인 + RAG 기반 | ✅ |
| Phase A~C | 데이터 보강 (TMDB/Kaggle/KOBIS/KMDb) | ✅ |
| Phase 2 | LLM 체인 + 프롬프트 | ✅ |
| Phase 3 | Chat Agent 그래프 (14노드) | ✅ |
| Phase 4 | 추천 엔진 서브그래프 (7노드) | ✅ |
| Phase 5~8 | SSE 최적화, Tools, 분석 에이전트, 테스트 | 미착수 |

## 5. 데이터 모델

### 5.1 ChatAgentState (TypedDict)

Chat Agent 그래프의 전체 상태를 정의하는 TypedDict. 각 노드가 이 상태를 읽고 쓴다.

```python
class ChatAgentState(TypedDict, total=False):
    # ── 입력 ──
    user_id: str                          # 사용자 ID
    session_id: str                       # 세션 ID
    current_input: str                    # 현재 메시지
    image_data: str | None                # base64 이미지 (선택)

    # ── context_loader 출력 ──
    user_profile: dict[str, Any]          # MySQL users 테이블
    watch_history: list[dict[str, Any]]   # MySQL watch_history + movies JOIN (최근 50건)
    messages: list[dict[str, str]]        # 누적 대화 메시지 [{role, content}]

    # ── image_analyzer 출력 ──
    image_analysis: ImageAnalysisResult   # VLM 이미지 분석 결과

    # ── intent_classifier 출력 ──
    intent: IntentResult                  # 의도 + 신뢰도

    # ── emotion_analyzer 출력 ──
    emotion: EmotionResult                # 감정 + 무드태그

    # ── preference_refiner 출력 ──
    preferences: ExtractedPreferences     # 7개 선호 필드
    needs_clarification: bool             # 정보 부족 여부
    turn_count: int                       # 대화 턴 수

    # ── question_generator 출력 ──
    follow_up_question: str               # 후속 질문

    # ── query_builder 출력 ──
    search_query: SearchQuery             # RAG 검색 쿼리

    # ── rag_retriever 출력 ──
    candidate_movies: list[CandidateMovie]  # 하이브리드 검색 결과 (10~15편)

    # ── recommendation_ranker 출력 ──
    ranked_movies: list[RankedMovie]      # 최종 추천 (Top 5)

    # ── response_formatter 출력 ──
    response: str                         # 최종 응답 텍스트

    # ── error_handler ──
    error: str | None                     # 에러 메시지
```

모든 노드는 아래 패턴을 따른다:

### 노드 1: context_loader

| 항목 | 내용 |
|---|---|
| **역할** | MySQL에서 유저 프로필/시청이력 로드, messages 구성 |
| **LLM** | 없음 |
| **입력** | user_id, session_id, current_input, messages |
| **출력** | user_profile, watch_history, messages, turn_count |
| **DB** | MySQL `users` 테이블 + `watch_history` JOIN `movies` (최근 50건) |
| **에러 복구** | DB 에러 시 빈 프로필/이력으로 계속 진행 |

**핵심 로직:**
1. 기존 messages에 현재 입력 추가 (`{"role": "user", "content": current_input}`)
2. 익명 사용자(user_id="") → 빈 기본값 반환
3. MySQL 쿼리: `SELECT * FROM users WHERE user_id = %s`
4. MySQL 쿼리: `SELECT wh.*, m.title, m.genres FROM watch_history wh JOIN movies m ... LIMIT 50`
5. turn_count = user 메시지 수 계산

## 8. 7개 추천 엔진 노드 상세

### 노드 1: cold_start_checker

```
시청이력 < 5편 → is_cold_start = True → popularity_fallback 경로
시청이력 ≥ 5편 → is_cold_start = False → CF+CBF 정상 경로
```

### 노드 2: collaborative_filter (CF)

**알고리즘:**
```
CF_score(u, m) = Σ sim(u,v) × r(v,m) / Σ |sim(u,v)|

- v: Top-50 유사 유저 (Redis Sorted Set)
- sim(u,v): 코사인 유사도 (사전 계산, Redis 캐시)
- r(v,m): 유저 v의 영화 m 평점 (Redis Hash)
```

**Redis 키 구조:**
- `cf:similar_users:{user_id}` — Sorted Set (유사도 점수)
- `cf:user_ratings:{user_id}` — Hash ({movie_id: rating})

**캐시 미스 처리:** `CF_DEFAULT_SCORE = 0.5`

### 노드 3: content_based_filter (CBF)

**감정 인식 가중치:**

| 컴포넌트 | 감정 있음 | 감정 없음 |
|---|---|---|
| 장르 Jaccard | 0.25 | 0.30 |
| 감독/배우 빈도 | 0.20 | 0.24 |
| 무드 매칭 | 0.15 | 0.00 |
| 키워드 매칭 | 0.15 | 0.16 |
| RRF 보존 | 0.25 | 0.30 |

**감독/배우 점수:**
```
dir_score = director_freq[movie.director] / sum(director_freq)
act_score = Σ(actor_freq[actor_i] / sum(actor_freq)) × 0.9^i  (top 3, decay)
crew_score = dir_score × 0.4 + act_score × 0.6
```

### 노드 4: hybrid_merger

**CF+CBF 가중치 조건표:**

| 조건 | CF | CBF |
|---|---|---|
| 시청 30편+ & 감정 없음 | 0.60 | 0.40 |
| 시청 30편+ & 감정 있음 | 0.50 | 0.50 |
| 시청 5~29편 & 감정 없음 | 0.40 | 0.60 |
| 시청 5~29편 & 감정 있음 | 0.30 | 0.70 |
| CF 캐시 미스 | 0.00 | 1.00 |
| CBF 전부 0 | 1.00 | 0.00 |
| 둘 다 실패 | RRF 점수 fallback |

### 노드 5: popularity_fallback (Cold Start 전용)

```
base_score = min(rating / 10, 1.0)
genre_boost = 0.1  (if 선호 장르 매칭)
mood_boost = 0.05  (if 무드 태그 매칭)
rrf_boost = rrf_score × 0.2
final = min(base + genre + mood + rrf, 1.0)
```

### 노드 6: diversity_reranker (MMR)

```
MMR_score(m) = λ × relevance(m) - (1-λ) × max(Jaccard(m.genres, s.genres))

λ = 0.7  (70% 관련성, 30% 다양성)
Greedy 선택: 1st = max(hybrid_score), 나머지 = max(MMR_score)
TOP_K = 5
```

### 노드 7: score_finalizer

- 상위 5편 선택
- 각 영화에 ScoreDetail 첨부 (cf/cbf/hybrid/genre_match/mood_match/similar_to)
- RankedMovie로 변환 (rank=1~5, explanation="" → explanation_generator에서 채움)

## 13. 추천 알고리즘 상세

### CF (Collaborative Filtering)

```
CF_score(u, m) = Σ sim(u,v) × r(v,m) / Σ |sim(u,v)|

- 270,888명 유저의 코사인 유사도 사전 계산
- Top-50 유사 유저 Redis 캐시 (Sorted Set)
- 26M 평점 데이터 기반
- Min-max 정규화 [0, 1]
```

### CBF (Content-Based Filtering)

```
CBF = w1×genre_jaccard + w2×crew_match + w3×mood_match + w4×keyword_match + w5×rrf_preserve

- genre_jaccard: Jaccard(movie.genres, liked_genres_from_history)
- crew_match: 감독(40%) + 배우 빈도(60%, decay 0.9^i)
- mood_match: |movie.mood_tags ∩ user.mood_tags| / |movie.mood_tags|
- keyword_match: Jaccard(movie.keywords, preference.keywords)
- rrf_preserve: movie.rrf_score / max_rrf (RAG 검색 순위 보존)
```

### MMR (Maximum Marginal Relevance)

```
MMR(m) = 0.7 × hybrid_score(m) - 0.3 × max(Jaccard(m.genres, selected.genres))

Greedy 선택:
1st: max(hybrid_score)
2nd~5th: max(MMR_score)

→ 장르 다양성 확보하면서 관련성 유지
```

## 21. 데이터 적재 전략 상세 분석

### 21.1 전체 적재 아키텍처 개요

몽글픽은 **5개 이기종 DB**에 동일한 영화 데이터를 각 DB의 특성에 맞게 변환하여 적재한다. 각 DB는 서로 다른 역할을 수행하며, 하나의 `MovieDocument` (80개 필드)에서 파생된다.

```
MovieDocument (80개 필드)
    │
    ├─→ [Qdrant]        벡터 검색 (4096차원 Cosine, 48+ payload 필드)
    ├─→ [Neo4j]         지식 그래프 (9종 노드, 20종 관계)
    ├─→ [Elasticsearch] BM25 텍스트 검색 (Nori 한국어 분석기, 60+ 필드)
    ├─→ [Redis]         CF 협업 필터링 캐시 (유사 유저 Top-50, 유저별 평점)
    └─→ [MySQL]         RDB 경량 참조 (30개 컬럼, 시청 이력 26M행)
```

### 21.3 임베딩 전략 (`embedder.py`)

#### Upstage Solar 임베딩

| 항목 | 값 |
|---|---|
| 모델 (문서) | `embedding-passage` |
| 모델 (쿼리) | `embedding-query` |
| 벡터 차원 | 4096 |
| 정규화 | magnitude=1 (코사인 유사도 = 내적) |
| API | OpenAI 호환 (base_url만 변경) |
| Rate Limit | 100 RPM / 300,000 TPM |

#### 배치 처리 전략

```python
# embedder.py — 핵심 로직
def embed_texts(texts: list[str], batch_size: int = 50) -> np.ndarray:
    for i in range(0, len(texts), batch_size):
        batch = texts[i : i + batch_size]
        response = client.embeddings.create(model="embedding-passage", input=batch)
        # 응답 순서가 입력 순서와 다를 수 있으므로 index 기준 정렬
        batch_embeddings = [
            item.embedding
            for item in sorted(response.data, key=lambda x: x.index)
        ]
        # Rate Limit: 100 RPM → 배치 간 0.7초 딜레이 (안전 마진 확보)
        if i + batch_size < len(texts):
            time.sleep(0.7)
```

**핵심 설계 결정**:
- **배치 크기 50**: TPM(300K) 제한 고려, 긴 텍스트일수록 토큰 소비가 많아 50건으로 보수적 설정
- **응답 정렬**: Upstage API가 입력 순서와 다른 순서로 응답할 수 있어 `index` 기준 정렬 필수
- **동기 API 사용**: `openai.OpenAI` (비동기 아님) — 임베딩은 배치 파이프라인에서만 사용하므로 동기로 충분
- **싱글턴 클라이언트**: 모듈 레벨 `_client` 변수로 재사용, API 키 인증 1회만 수행

#### UNWIND 배치 패턴

Neo4j에 대량 데이터를 적재할 때 핵심이 되는 패턴이다.

```cypher
-- _batch_execute()가 사용하는 UNWIND 패턴
-- 500건씩 배치로 나누어 실행 (Neo4j 메모리 부족 방지)
UNWIND $batch AS m
MERGE (movie:Movie {id: m.id})
SET movie.title = m.title,
    movie.release_year = m.release_year,
    ...
```

**MERGE vs CREATE**: 항상 `MERGE`를 사용하여 중복 방지. 동일 ID/name의 노드가 이미 존재하면 업데이트, 없으면 생성.

**CASE WHEN 조건부 업데이트** (Person 노드):
```cypher
-- 기존 값이 있으면 유지 (더 상세한 정보 우선)
SET person.person_id = CASE WHEN p.person_id > 0 THEN p.person_id ELSE person.person_id END,
    person.profile_path = CASE WHEN p.profile_path <> '' THEN p.profile_path ELSE person.profile_path END
```

**MATCH 기반 관계 생성**: 양쪽 노드가 모두 존재할 때만 관계를 생성한다.
```cypher
MATCH (m1:Movie {id: s.movie_id})  -- 없으면 행 자체가 무시됨
MATCH (m2:Movie {id: s.similar_id})
MERGE (m1)-[:SIMILAR_TO]->(m2)
```

#### 핵심 설계 결정

| 결정 | 이유 |
|---|---|
| **배치 500건** | Neo4j 트랜잭션 메모리 제한 고려, 500건이 안정적 |
| **UNWIND 패턴** | 개별 CREATE보다 네트워크 왕복 대폭 감소 (500배) |
| **MERGE 사용** | 멱등성 보장 — 동일 파이프라인 재실행 시에도 중복 없음 |

### 21.7 Redis CF 캐시 적재 전략 (`cf_builder.py`)

#### 전체 흐름

```
Kaggle ratings.csv (26M행)
    │
    ├─ [1] links.csv로 MovieLens ID → TMDB ID 매핑
    │
    ├─ [2] CSR(Compressed Sparse Row) 스파스 매트릭스 구축
    │      shape: (270,888 유저) × (44,872 영화)
    │      nnz: ~26M (비영 원소)
    │
    ├─ [3] 유사도 계산 (1,000명 청크)
    │      ├─ 전체 유저를 한번에 계산하면 O(N^2) = 270K² = ~73GB 메모리 필요
    │      ├─ → 1,000명씩 청크로 분할: 1,000 × 270,888 행렬만 메모리에 유지
    │      ├─ cosine_similarity(chunk_matrix, full_matrix)
    │      ├─ 자기 자신과의 유사도 = -1로 설정하여 제외
    │      └─ np.argsort()[-top_k:][::-1] → Top-50 유사 유저 추출
    │
    └─ [4] Redis 캐시 저장 (5,000건 배치 pipeline)
```

#### Redis 키 구조

```
cf:similar_users:{user_id}      → Sorted Set (유사 유저 Top-50, score=유사도)
                                   TTL: 24시간 (86,400초)
                                   예: ZADD cf:similar_users:123 0.95 "456" 0.87 "789" ...

cf:user_ratings:{user_id}       → Hash (영화별 평점)
                                   TTL: 24시간
                                   예: HSET cf:user_ratings:123 "157336" "4.5" "550" "3.0" ...

cf:movie_avg_rating:{movie_id}  → String (영화 평균 평점)
                                   TTL: 24시간
                                   예: SET cf:movie_avg_rating:157336 "4.2"

cf:matrix_version               → String (매트릭스 빌드 타임스탬프)
                                   TTL: 없음
                                   예: "20260301_1430"
```

#### Redis 배치 저장 전략

```python
# 270K 유저를 한 pipeline에 넣으면 Redis OOM 발생 → 5,000건씩 분할
batch_size = 5000

for i in range(0, len(user_items), batch_size):
    batch = user_items[i : i + batch_size]
    pipe = redis.pipeline()  # 커맨드를 모아서 한번에 전송 (네트워크 왕복 감소)
    for uid, sim_list in batch:
        pipe.delete(key)      # 기존 데이터 삭제 (갱신 보장)
        pipe.zadd(key, mapping)
        pipe.expire(key, 86400)
    await pipe.execute()
```

#### 핵심 설계 결정

| 결정 | 이유 |
|---|---|
| **CSR 스파스 매트릭스** | 26M개 평점을 270K×45K 밀집 행렬로 저장하면 ~90GB → 스파스로 ~400MB |
| **1,000명 청크** | 전체 유사도 O(N²) 계산 불가 → 청크 분할로 메모리 사용량 제한 |
| **Top-50 유사 유저** | 유사도 > 0인 유저만 저장 (0 이하 제외), 50명이면 추천 품질 충분 |
| **Sorted Set** | 유사도 score 기준 범위 쿼리 지원 (ZRANGEBYSCORE) |
| **Hash** | O(1) 평점 조회 (특정 영화 평점 확인) |
| **Pipeline 배치** | 5,000건씩 묶어 전송 → 네트워크 왕복 270K/5K = 54회로 감소 |
| **TTL 24시간** | 일일 갱신 주기에 맞춰 자동 만료 (maxmemory-policy allkeys-lru와 함께 사용) |
| **기존 키 삭제 후 재적재** | Sorted Set의 부분 업데이트보다 전체 교체가 일관성 보장 |

#### CF 점수 계산 (추천 엔진에서 사용)

```
CF_score(u, m) = Σ sim(u,v) × r(v,m) / Σ |sim(u,v)|

v: cf:similar_users:{user_id}에서 조회한 Top-50 유사 유저 중 영화 m을 평가한 유저
r(v,m): cf:user_ratings:{v}에서 조회한 유저 v의 영화 m 평점
sim(u,v): Sorted Set score (코사인 유사도)
```
